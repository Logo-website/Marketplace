# Документ защиты стека (architecture decisions)

Дата: 2026-06-15
Статус: финал фазы P10 плана `plans/2026-06-13-rational-stack-defense.md`.

## Зачем этот документ

Цель проекта - показать, что я владею всем стеком, **использую его рационально** и понимаю безопасность.
Стек учебно-портфельный, стадия pre-launch (0 пользователей, seed-каталог). Поэтому к каждому
компоненту - не «он крутой», а защита по трём вопросам:

1. **Какую конкретную боль он решает?**
2. **Реально ли подключён end-to-end** (данные текут от события до результата)?
3. **Почему именно он, а не проще?**

И для каждого тяжёлого компонента - **метрика-триггер**: при каком объёме он перестаёт быть
избыточным. Сейчас многое избыточно для 0 пользователей - это честно зафиксировано, а не спрятано.

## Таблица защиты компонентов

### PostgreSQL - основная OLTP-база

- **Боль.** Транзакционное хранение каталога, пользователей, заказов с гарантией целостности и
  конкурентным списанием стока.
- **End-to-end.** Весь CRUD каталога/профилей; заказы с атомарным списанием стока через
  `select_for_update(of=('self',))` ([orders/models.py](../backend/apps/orders/models.py)),
  идемпотентная отмена с возвратом стока; денормализованный `Product.rating`/`reviews_count`
  ([products/models.py](../backend/apps/products/models.py)), индексы на горячие поля.
- **Почему не проще.** Нужны ACID-транзакции, FK-целостность и блокировки строк под конкурентный
  сток. SQLite не держит конкурентную запись; «что-то сложнее» не требуется.
- **Метрика-триггер.** Шардинг/реплики - при росте записи сверх одного узла (десятки тысяч
  заказов/день). Сейчас single-node достаточно.

### Redis - кэш + корзина

- **Боль.** (1) Эфемерное состояние корзины вне БД; (2) кэш горячих, редко меняющихся ответов,
  чтобы не бить Postgres на каждый запрос.
- **End-to-end.** Корзина в Redis db/0 ([cart/cart.py](../backend/apps/cart/cart.py), ленивый
  `get_redis`); Django `CACHES` на Redis db/1 ([config/settings.py](../backend/config/settings.py)):
  кэшируются категории (TTL 1ч) и карточка товара (TTL 5мин) с инвалидацией сигналами на `Product`
  ([products/caching.py](../backend/apps/products/caching.py), [products/signals.py](../backend/apps/products/signals.py)).
- **Почему не проще.** Локальная in-process память не работает при 4 gunicorn-воркерах (каждый свой
  кэш, рассинхрон инвалидации). Redis - общий слой для всех воркеров.
- **Метрика-триггер.** Уже оправдан (4 воркера в [docker-compose.yml](../docker-compose.yml)). Отдельный
  Redis-кластер - при кэше сверх объёма памяти одного узла.

### Elasticsearch - поиск

- **Боль.** Главная бизнес-боль «хаос выбора»: поиск с опечатками, морфологией русского, фасетами
  со счётчиками и автокомплитом. Это отличие от Lamoda и ядро ценности.
- **End-to-end.** Индексация ([products/search.py](../backend/apps/products/search.py), ленивый `get_es`),
  `search_products` отдаёт `{ids, total, facets}` (post_filter + per-facet filtered aggs), автокомплит
  через `match_phrase_prefix`, пагинация, согласованный фильтр по `category_id`. Фронт - сайдбар фасетов
  + подсказки. Graceful: ES лёг -> пустая выдача, не 500.
- **Почему не проще.** Postgres FTS даёт полнотекст, но фасетные агрегации со счётчиками + fuzzy +
  русская морфология из коробки - это ES. Trigram-индексы Postgres покрыли бы опечатки, но не дают
  дешёвых фасет-счётчиков.
- **Метрика-триггер.** При <~10k товаров Postgres trigram + ручные агрегации справились бы. ES
  оправдан морфологией и как честная демонстрация фасетного поиска. Кластер - от миллионов
  документов / десятков QPS поиска.

### RabbitMQ + Celery - очередь задач

- **Боль.** Вынести медленные/ненадёжные побочки из HTTP-пути, чтобы gunicorn-воркер не висел на
  внешнем сервисе (Resend, ClickHouse) и не деградировал под нагрузкой.
- **End-to-end.** Письма OTP (`send_otp_email`), аналитика (`track_event`,
  [products/tasks.py](../backend/apps/products/tasks.py)), Kafka-публикация
  (`publish_order_event`, [orders/tasks.py](../backend/apps/orders/tasks.py)), периодический пересчёт
  матрицы ко-покупок (`build_copurchase_matrix` + `beat`). Commit-safety: все побочки заказа
  диспатчатся через `transaction.on_commit(...)` ([orders/views.py](../backend/apps/orders/views.py)) -
  воркер не стартует задачу до коммита транзакции.
- **Почему не проще.** Синхронный вызов внешнего сервиса в HTTP-пути - вектор отказа (S8): тормозит
  сервис - занят воркер. Celery развязывает.
- **Метрика-триггер.** Уже оправдан (есть реально асинхронные задачи). Несколько воркеров/приоритетные
  очереди - при росте потока писем и фоновых пересчётов.

### Kafka - поток событий для вещания

- **Боль.** Доставка событий заказа (`created`/`status_changed`) в real-time канал (WebSocket) с
  возможностью независимого вещания нескольким потребителям.
- **End-to-end.** `publish_order_event` (Celery) -> Kafka -> consumer в [node_service/server.js](../node_service/server.js)
  -> push авторизованному клиенту по WS. Проверено e2e на живом стеке (P9): Kafka-событие доставлено
  владельцу заказа.
- **Почему не проще / overlap с RabbitMQ.** См. раздел «Пары-overlap» ниже.
- **Метрика-триггер.** Kafka незаменим при настоящем event-streaming: несколько независимых
  консьюмеров одного потока, реплей с оффсета, аудит-лог. На 0 пользователей - избыточен, держится
  как демонстрация понимания разницы «очередь задач vs лог событий».

### ClickHouse - OLAP-аналитика

- **Боль.** (1) Аналитика продавца (просмотры/покупки) без нагрузки на OLTP-Postgres; (2) тяжёлая
  агрегация матрицы ко-покупок (self-join событий по парам товаров).
- **End-to-end.** `write_event` пишет view/purchase с реальным `order_id`
  ([services/clickhouse_service.py](../backend/services/clickhouse_service.py)); `get_copurchase_pairs`
  считает пары `GROUP BY order_id`; матрица -> JSON-volume -> C++. Статистика продавца читается из
  ClickHouse.
- **Почему не проще / overlap с Postgres.** См. раздел «Пары-overlap» ниже.
- **Метрика-триггер.** При >~10М событий аналитики строчный Postgres на агрегациях деградирует -
  колоночный ClickHouse держит. На seed-объёме Postgres посчитал бы то же; ClickHouse оправдан как
  правильный инструмент под append-only аналитику и демонстрация OLAP/OLTP-разделения.

### node.js + WebSocket - real-time последняя миля

- **Боль.** Живые статусы заказа у пользователя без поллинга HTTP.
- **End-to-end.** Kafka consumer -> JWT-валидация (S5) -> push клиенту;
  фронт ([notificationStore.js](../frontend/src/store/notificationStore.js)) подключается после логина,
  реконнект, закрытие при логауте, тосты. Адрес WS через `VITE_WS_URL`.
- **Почему не проще.** Долгоживущие WS-соединения держать на синхронном gunicorn неэффективно
  (занимает воркеры). Можно было Django Channels, но отдельный node-сервис на нативном event-loop -
  честная полиглот-демонстрация и разгрузка backend.
- **Безопасность.** node проверяет JWT общим с Django `SECRET_KEY` (`HS256`, `token_type=='access'`),
  `user_id` берётся ИЗ токена, токен - в первом ws-сообщении (не в query - не течёт в логи).
- **Метрика-триггер.** Несколько node-инстансов за load-balancer + sticky/Redis pub-sub - при тысячах
  одновременных соединений.

### C++ рекомендатель - in-memory lookup ко-покупок

- **Боль (честно).** На нашем масштабе lookup топ-N сопутствующих товаров мог бы делать Django+Redis.
  C++ оставлен как **осознанная демонстрация полиглот-интеграции**, подключённая end-to-end
  по-настоящему, а не как заглушка.
- **End-to-end.** ClickHouse считает матрицу (Celery) -> текстовый формат в общий volume
  (`/data/copurchase_matrix.txt`) -> C++ ([cpp_service/main.cpp](../cpp_service/main.cpp)) грузит в память
  и перечитывает по mtime -> `RecommendationsView` зовёт C++ по HTTP -> маппинг на активные товары ->
  graceful fallback на популярное по категории. Безопасный парсинг (только цифры, без `std::stoi`-исключений).
- **Почему именно C++.** Тяжёлая агрегация - в ClickHouse, отдача - чистый in-memory lookup в C++ без
  ClickHouse-драйвера внутри. На больших объёмах C++ дал бы суб-миллисекундную латентность.
- **Метрика-триггер.** Оправдан при миллионах товаров и требовании латентности <1мс на запрос
  рекомендаций. На seed-каталоге - демонстрация. **Альтернатива при мешании:** понизить до явного «демо»
  в README и звать ClickHouse напрямую - тоже защитимо.

## Пары-overlap (явно зафиксировано)

### Kafka vs RabbitMQ - зачем два брокера

Разные семантики, не дублирование:

| | RabbitMQ (+ Celery) | Kafka |
|---|---|---|
| Модель | Очередь задач (work queue) | Лог событий (pub/sub, append-only) |
| Доставка | Одна задача - один консьюмер, ack/retry | Broadcast, несколько независимых консьюмеров |
| Семантика | «Сделай работу» (письмо, пересчёт) | «Случилось событие» (заказ создан) |
| Реплей | Нет (задача выполнена и ушла) | Есть (чтение с оффсета) |
| У нас | Письма OTP, аналитика, пересчёт матрицы | События заказа -> WebSocket-вещание |

**Честно:** на 0 пользователей один брокер справился бы с обеими ролями. Два держим как
демонстрацию понимания разницы «очередь задач vs поток событий». Триггер, когда Kafka становится
незаменим: появление второго независимого консьюмера того же потока событий (например, аналитика +
WS одновременно слушают `order.*`) или потребность в реплее.

### ClickHouse vs Postgres - зачем две БД

| | PostgreSQL | ClickHouse |
|---|---|---|
| Тип | OLTP, строчный | OLAP, колоночный |
| Нагрузка | Точечные чтения/записи, транзакции | Агрегации по миллионам строк, append-only |
| У нас | Каталог, заказы, пользователи, сток | События view/purchase, матрица ко-покупок |
| Запись | Транзакционная, единичная | Пакетная, неизменяемая |

**Честно:** на seed-объёме Postgres посчитал бы ту же матрицу ко-покупок. ClickHouse - правильный
инструмент под append-only аналитику и демонстрация OLTP/OLAP-разделения. Триггер: >~10М событий,
где строчные агрегации Postgres деградируют.

## Что осознанно НЕ делаем (и почему это защитимо)

- **Keyset/cursor-пагинация.** Offset достаточен на seed-каталоге; keyset нужен на глубоких
  страницах больших таблиц. (P6 скип.)
- **Первая страница каталога не кэшируется** (в отличие от категорий/карточки). Список меняется чаще
  (сток, новые товары, сортировки), инвалидация сложнее, выигрыш на seed-каталоге мал. (P6b скип.)
- **ML-рекомендации / эмбеддинги** (как у WB). Вне масштаба; item-to-item по ко-покупкам (Amazon,
  1998) честнее и понятнее для портфолио.
- **Платёжный шлюз.** Заказы без оплаты - сознательное упрощение pre-launch (зафиксировано в README).
- **Микросервисное дробление.** Монолит Django оправдан на стадии 0 пользователей; дробить рано.
- **Кластерные ClickHouse / ES / Kafka.** Single-node под учебный объём.
- **Суб-заказы на продавца (статус на `OrderItem`).** Архитектурно правильно для маркетплейса (WB/Ozon),
  но это отдельная фича на недели, не security-фикс. P2 закрыл дыру уточнённым вариантом Б (продавец
  ведёт только полностью свои заказы). Триггер эволюции: появление реальных мультивендорных заказов.

## Безопасность: осознанные упрощения (из security review)

Полный трекер - `plans/2026-06-13-security-review.md`. Critical/High (S1-S5) закрыты и проверены.
Здесь - принятые упрощения и их обоснование:

- **S7 - JWT в `localStorage` + `Authorization: Bearer`.** Осознанное упрощение: к API CSRF неприменим,
  httpOnly-cookie - отдельная фича, не оправдана на pre-launch. Риск XSS-кражи токена назван в README.
  CSRF-настройки остаются только под контур Django admin / DRF browsable API.
- **S13 (Low) - тайминг-энумерация при сбросе пароля.** Ответ одинаков для существующего и
  несуществующего email (не раскрывает существование), но при существующем идёт отправка письма ->
  тайминг чуть отличается. **Осознанный скип:** Low-риск, выравнивание тайминга (отправка в Celery в
  обоих случаях / фиктивная задержка) не оправдано на учебном проекте. Зафиксировано здесь как
  принятое.
- **S15 (Low) - `/api/docs/` открыт авторизованным.** `DEFAULT_PERMISSION = IsAuthenticated`, схема
  видна любому залогиненному. **Норма для учебного проекта** - схема API не секрет; зафиксировано как
  осознанное. На проде с реальными данными - закрыть до staff-only.
- **S17 (Medium) - email продавца раскрывался анонимам. ЗАКРЫТО (P10).** `ProductSerializer.seller_name`
  больше не отдаёт `seller.email`: добавлено non-PII поле `User.shop_name` (миграция
  `0004_user_shop_name` с бэкофиллом из username для продавцов); сериализатор отдаёт
  `shop_name or username`, никогда email ([products/serializers.py](../backend/apps/products/serializers.py)).
  Фикс закрывает все эндпоинты с этим сериализатором (каталог, карточка, поиск, рекомендации). Тесты
  `test_seller_email_not_exposed_in_catalog`, `test_seller_shop_name_used_when_set`.
- **Симметричный JWT для node (P9).** node только проверяет токен общим `SECRET_KEY`, не подписывает;
  оба сервиса first-party. Асимметрия (RS256, у node только публичный ключ) строже, но избыточна на
  этой стадии.

## Итог

Стек оставлен целиком и защитим: каждый компонент отвечает на три вопроса, у каждого тяжёлого -
метрика-триггер избыточности, пары-overlap (Kafka/RabbitMQ, ClickHouse/Postgres) разведены явно.
Где компонент избыточен на 0 пользователей - это сказано честно, а не спрятано. Безопасность: все
Critical/High закрыты и проверены, оставшиеся Low - осознанные упрощения, зафиксированные выше.
